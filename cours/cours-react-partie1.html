01

Composants Fonctionnels

Qu'est-ce qu'un composant ?
Imagine que tu construis une maison. Tu ne construis pas la maison d'un seul coup — tu assembles des briques, des fenêtres, des portes. En React, c'est exactement pareil : tu découpes ton interface en petits blocs réutilisables qu'on appelle des composants.

Un composant React, c'est simplement une fonction JavaScript qui retourne du JSX. Le JSX ressemble à du HTML mais c'est du JavaScript déguisé.

Pourquoi les composants ? Parce que si tu as une carte d'atelier à afficher 50 fois, tu l'écris une seule fois et tu la réutilises 50 fois. Si tu veux changer son apparence, tu changes à un seul endroit.

  UNE PAGE = UN ASSEMBLAGE DE COMPOSANTS

  ┌─────────────────────────────────────────────────────────┐
  │  App.jsx   (composant racine — le point de départ)      │
  │                                                         │
  │  ┌──────────────────────────────────────────────────┐   │
  │  │   <Navigation />                                  │   │
  │  └──────────────────────────────────────────────────┘   │
  │                                                         │
  │  ┌─────────────────────────────────────────────────┐    │
  │  │   Page en cours (ex: <TableauDeBord />)          │    │
  │  │                                                 │    │
  │  │  ┌─────────────┐  ┌─────────────┐              │    │
  │  │  │ <Carte />    │  │ <Carte />    │  ...         │    │
  │  │  │             │  │             │              │    │
  │  │  │ <Bouton />  │  │ <Bouton />  │              │    │
  │  │  └─────────────┘  └─────────────┘              │    │
  │  └─────────────────────────────────────────────────┘    │
  └─────────────────────────────────────────────────────────┘

  Chaque boîte = une fonction JS dans un fichier .jsx séparé
  Tous ces composants s'emboîtent comme des poupées russes

La règle fondamentale : majuscule obligatoire

React fait la différence entre une balise HTML native (<div>, <p>, <button>...) et un composant React grâce à la première lettre :

  • <div> minuscule → HTML standard, le navigateur le connaît
  • <Carte> Majuscule → composant React, c'est notre fonction
Carte.jsx — un composant complet expliqué ligne par ligne 
// ─────────────────────────────────────────────────────────────
// LIGNE 1 : on importe React si on utilise des hooks (useState,
//           useEffect...). En React 18, cet import est optionnel
//           pour le JSX pur, mais on l'ajoute dès qu'on a des hooks.
// ─────────────────────────────────────────────────────────────

// Pour l'instant on n'a pas de hooks, pas d'import nécessaire.

// ─────────────────────────────────────────────────────────────
// "function" : mot-clé JS pour déclarer une fonction.
// "Carte"    : nom du composant. DOIT commencer par une Majuscule.
//              Sinon React pense que c'est une balise HTML.
// "()"        : pas de paramètres pour l'instant (on verra les props).
// ─────────────────────────────────────────────────────────────
function Carte() {

  // ──────────────────────────────────────────────────
  // "return" : ce que le composant affiche à l'écran.
  //            Tout ce qui est à l'intérieur du return
  //            devient du HTML dans le navigateur.
  // ──────────────────────────────────────────────────
  return (
    // Les parenthèses () après return permettent d'écrire
    // du JSX sur plusieurs lignes sans erreur.
    //
    // RÈGLE JSX : il ne peut y avoir qu'UN SEUL élément
    // racine. Ici c'est le <div> extérieur qui enveloppe tout.
    // Si tu as <div> et <p> côte à côte sans parent commun → erreur.

    <div className="carte">
      {/* className remplace "class" du HTML.
           En JavaScript, "class" est un mot réservé (pour les
           classes ES6), donc JSX utilise className à la place.
           Dans le navigateur, ça redevient class="carte" */}

      <h2>React Avancé</h2>

      <p>Apprenez les patterns avancés de React.</p>

      {/* Les accolades {} dans le JSX permettent d'insérer
           du JavaScript. Ici on calcule directement */}
      <p>Places restantes : {30 - 22}</p>

    </div>
  )
}

// ──────────────────────────────────────────────────────────────
// "export default" : rend ce composant disponible pour d'autres
//                    fichiers. Sans ce mot-clé, personne ne peut
//                    utiliser Carte en dehors de ce fichier.
// ──────────────────────────────────────────────────────────────
export default Carte
App.jsx — utiliser le composant 
// "import" : on importe le composant Carte depuis le fichier Carte.jsx.
// Le chemin commence par "./" qui signifie "dans le même dossier".
// Pas besoin d'écrire ".jsx", Vite le trouve tout seul.
import Carte from './Carte'

function App() {
  return (
    <div>
      {/* On utilise Carte exactement comme une balise HTML.
           Les balises auto-fermantes (/>) sont valides en JSX. */}
      <Carte />
      <Carte />  {/* Le même composant réutilisé — c'est tout l'intérêt */}
      <Carte />
    </div>
  )
}

export default App
💡
JSX n'est pas du HTML. Le navigateur ne comprend pas le JSX. Vite (ou Webpack) le convertit en JavaScript pur avant d'envoyer le code au navigateur. <h2>Bonjour</h2> devient React.createElement('h2', null, 'Bonjour'). Le JSX est juste une syntaxe plus lisible.
02

Props — Passer des données

Le problème : Notre composant <Carte /> affiche toujours le même texte "React Avancé". C'est inutile — une carte identique pour 50 ateliers différents n'a aucun sens.

La solution : les props. Les props (propriétés) sont les arguments d'un composant. Exactement comme une fonction JavaScript reçoit des paramètres, un composant React reçoit des props.

Règle absolue : les props sont en lecture seule. L'enfant ne peut pas les modifier. Les données coulent toujours du parent vers l'enfant — jamais l'inverse. C'est ce qu'on appelle le flux unidirectionnel.

  PARENT envoie des données vers les ENFANTS via les props :

  <Carte titre="React Avancé" inscrits={28} actif={true} />
           │                   │              │
           │                   │              └─ boolean : accolades obligatoires
           │                   └─ nombre : accolades obligatoires
           └─ string : guillemets suffisent (ou accolades, les deux marchent)

  DANS LE COMPOSANT ENFANT :
  ┌─────────────────────────────────────────────────────────┐
  │  function Carte(props) {                                │
  │  //                 ↑                                   │
  │  //  "props" est un objet qui contient tout :           │
  │  //  props.titre   → "React Avancé"                     │
  │  //  props.inscrits → 28                                │
  │  //  props.actif   → true                               │
  │  }                                                      │
  └─────────────────────────────────────────────────────────┘

  VERSION AMÉLIORÉE — destructuration :
  ┌─────────────────────────────────────────────────────────┐
  │  function Carte({ titre, inscrits, actif }) {           │
  │  //              ↑ on extrait directement les props     │
  │  //              ↑ plus besoin d'écrire props.titre     │
  │  }                                                      │
  └─────────────────────────────────────────────────────────┘
Carte.jsx — composant qui reçoit des props 
// ────────────────────────────────────────────────────────────────
// On déclare les props attendues entre accolades dans les paramètres.
// C'est de la "destructuration" JavaScript :
//   function Carte({ titre, inscrits })
// est équivalent à :
//   function Carte(props) {
//     const titre    = props.titre
//     const inscrits = props.inscrits
//   }
// La destructuration est juste plus courte et plus lisible.
// ────────────────────────────────────────────────────────────────
function Carte({ titre, description, inscrits, maxPlaces, children }) {

  return (
    <div className="carte">

      {/* Les accolades {} dans le JSX = "ici c'est du JavaScript".
           On affiche la valeur de la variable "titre". */}
      <h2>{ titre }</h2>

      <p>{ description }</p>

      {/* On peut faire du JavaScript dans les accolades :
           calcul, comparaison, appel de méthode... */}
      <p>
        { inscrits } / { maxPlaces } inscrits
        ({ Math.round((inscrits / maxPlaces) * 100) }%)
      </p>

      {/* "children" est une prop spéciale.
           Elle contient tout ce qu'on met ENTRE les balises
           du composant : <Carte>...ici...</Carte>
           Si rien n'est passé, children vaut undefined. */}
      <div>{ children }</div>

    </div>
  )
}

export default Carte
App.jsx — passer des props au composant 
import Carte from './Carte'

function App() {
  return (
    <div>

      {/* On passe les props comme des attributs HTML.
           titre="..." → string entre guillemets
           inscrits={28} → nombre entre accolades
           Les accolades sont obligatoires pour tout ce qui
           n'est pas une chaîne de caractères simple. */}
      <Carte
        titre="React Avancé"
        description="Maîtrisez les hooks et patterns avancés"
        inscrits={28}

      >
        {/* Ceci ira dans la prop "children" de Carte */}
        <button>S'inscrire</button>
      </Carte>

      {/* Même composant, données différentes → affichage différent */}
      <Carte

        inscrits={15}
        maxPlaces={20}

⚠️
Ne jamais modifier une prop dans l'enfant. Si tu fais titre = "nouveau titre" à l'intérieur de Carte, React ne le détectera pas et l'interface ne se mettra pas à jour. Les props sont en lecture seule. Pour des données qui changent, il faut le state (useState).
03

State avec useState

Pourquoi le state existe-t-il ?
Les props viennent du parent et ne changent pas dans l'enfant. Mais une interface web a besoin de données qui changent en réponse aux actions de l'utilisateur : un compteur qui monte, un formulaire qu'on remplit, une modal qui s'ouvre...

useState est le hook qui permet à un composant de mémoriser une valeur entre les rendus. Si tu utilises une variable JavaScript normale (let compteur = 0), elle est réinitialisée à chaque fois que React re-affiche le composant. useState, lui, conserve la valeur.

Et surtout : quand tu modifies le state avec le setter, React re-affiche automatiquement le composant avec la nouvelle valeur. C'est le mécanisme fondamental de React.

  useState retourne un tableau de 2 éléments.
  On les extrait par destructuration :

  const  [  valeur  ,   setValeur  ]  =  useState(  0  )
              │              │                       │
              │              │                       └─ valeur initiale (au premier affichage)
              │              └─ fonction pour CHANGER la valeur
              └─ la valeur actuelle (ce qu'on affiche dans le JSX)


  LE CYCLE COMPLET (ce qui se passe quand on clique) :

  ┌─────────────────────────────────────────────────────────────┐
  │                                                             │
  │  1. Composant s'affiche        valeur = 0                   │
  │           │                                                 │
  │           ▼                                                 │
  │  2. Utilisateur clique sur le bouton "+"                    │
  │           │                                                 │
  │           ▼                                                 │
  │  3. onClick appelle  setValeur(valeur + 1)                   │
  │           │            └─ on demande à React de             │
  │           │               mettre valeur à 1                 │
  │           ▼                                                 │
  │  4. React planifie un nouveau rendu                         │
  │           │                                                 │
  │           ▼                                                 │
  │  5. React appelle à nouveau la fonction du composant        │
  │           │                                                 │
  │           ▼                                                 │
  │  6. Cette fois, useState(0) retourne [1, setValeur]         │
  │           │         ↑ React a mémorisé la nouvelle valeur   │
  │           ▼                                                 │
  │  7. Le JSX s'affiche avec valeur = 1  ✓                      │
  │           │                                                 │
  │           └────────────── retour à l'étape 1               │
  └─────────────────────────────────────────────────────────────┘
Compteur.jsx — useState avec un nombre 
// useState vient de la librairie React.
// On l'importe avec les accolades (import nommé).
// La librairie React exporte beaucoup de choses : useState,
// useEffect, useRef, etc. On n'importe que ce dont on a besoin.
import { useState } from 'react'

function Compteur() {

  // ────────────────────────────────────────────────────────────
  // DÉCLARATION DU STATE :
  //
  // useState(0)   → on initialise le state à 0
  // compte        → c'est la valeur actuelle (0 au départ)
  // setCompte     → c'est la fonction pour changer "compte"
  //
  // Convention : le setter s'appelle "set" + nomDuState en camelCase.
  // Ce n'est pas obligatoire, mais c'est la convention universelle.
  // ────────────────────────────────────────────────────────────
  const [compte, setCompte] = useState(0)

  return (
    <div>

      {/* On affiche la valeur du state dans le JSX.
           À chaque changement de "compte", ce texte se met à jour. */}
      <p>Valeur actuelle : { compte }</p>

      {/* onClick reçoit une fonction fléchée.
           Cette fonction appelle setCompte avec la nouvelle valeur.
           On utilise "compte + 1" pour incrémenter l'actuel. */}
      <button onClick={() => setCompte(compte + 1)}>
        + 1
      </button>

      {/* Forme alternative avec une fonction séparée.
           Utile quand la logique est plus complexe. */}
      <button onClick={() => setCompte(compte - 1)}>
        - 1
      </button>

      {/* setCompte(0) remet le state à 0, peu importe sa valeur */}
      <button onClick={() => setCompte(0)}>
        Réinitialiser
      </button>

    </div>
  )
}

export default Compteur
FormulaireSimple.jsx — useState avec un objet 
import { useState } from 'react'

function FormulaireSimple() {

  // On peut stocker n'importe quel type de données dans le state :
  // string, number, boolean, array, object...
  // Ici on stocke un objet avec plusieurs champs de formulaire.
  // C'est pratique pour garder tous les champs groupés.
  const [formData, setFormData] = useState({
    nom:    '',   // chaîne vide = champ vide au départ
    email:  '',
    niveau: 'beginner'
  })

  const changerNom = (e) => {
    // "e" (ou "event") est l'objet événement.
    // e.target est l'élément HTML qui a déclenché l'événement (l'input).
    // e.target.value est la valeur tapée dans l'input.

    // TRÈS IMPORTANT : le spread operator "..." copie tous les
    // champs existants de formData. Ensuite on ÉCRASE seulement
    // le champ "nom" avec la nouvelle valeur.
    // Sans le "...", on perdrait email et niveau !
    setFormData({ ...formData, nom: e.target.value })
  }

  return (
    <div>
      <input
        type="text"
        value={formData.nom}      {/* React contrôle la valeur affichée */}
        onChange={changerNom}     {/* On met à jour le state à chaque frappe */}
        placeholder="Votre nom"
      />

      {/* Affichage en temps réel — se met à jour à chaque frappe */}
      <p>Tu as tapé : { formData.nom }</p>
    </div>
  )
}
🚫
Erreur fatale — ne JAMAIS modifier le state directement :
compte = compte + 1 → ne fonctionne pas, React ne le détecte pas.
formData.nom = "Jean" → ne fonctionne pas.
Toujours utiliser la fonction setter : setCompte(compte + 1), setFormData({...}).
04

useEffect — Effets de bord

C'est quoi un "effet de bord" ?
Lors d'un rendu, React fait UNE chose : appeler ta fonction composant et afficher ce qu'elle retourne. Tout le reste — appeler une API, modifier le titre de la page, créer un timer, écouter un événement — est considéré comme un "effet de bord" car ça dépasse le simple affichage.

useEffect te donne un endroit sûr pour exécuter ces actions, après que React a terminé d'afficher le composant. Ça évite de perturber le rendu.

  SYNTAXE :
  useEffect(  fonction  ,  [tableau de dépendances]  )
                  │                    │
                  │                    └── QUAND ré-exécuter la fonction ?
                  └── QUOI faire (ton effet)


  LES 3 CAS DU TABLEAU DE DÉPENDANCES :

  ┌──────────────────────────────────────────────────────────────┐
  │                                                              │
  │  useEffect(() => { ... })                                    │
  │  └─ Pas de tableau → se déclenche après CHAQUE rendu.       │
  │     Presque jamais utile. Attention aux boucles infinies !  │
  │                                                              │
  │  useEffect(() => { ... }, [])                                │
  │  └─ Tableau vide → se déclenche UNE SEULE FOIS              │
  │     au montage (quand le composant apparaît pour            │
  │     la première fois). Parfait pour les appels API.         │
  │                                                              │
  │  useEffect(() => { ... }, [userId, filtre])                  │
  │  └─ Tableau avec valeurs → se déclenche au montage          │
  │     ET à chaque fois que userId OU filtre change.           │
  │                                                              │
  └──────────────────────────────────────────────────────────────┘


  CYCLE DE VIE :

  Composant            useEffect(fn, [])     useEffect(fn, [val])
  s'affiche    ──────→  exécute fn           exécute fn
  val change   ──────→  (rien)               exécute fn
  composant    ──────→  cleanup              cleanup
  disparaît
ListeAteliers.jsx — fetch API au montage 
import { useState, useEffect } from 'react'

function ListeAteliers() {

  // 3 states pour gérer les 3 états d'un appel API :
  // - on attend      → chargement = true
  // - succès         → ateliers contient les données
  // - erreur         → erreur contient le message
  const [ateliers,    setAteliers]    = useState([])
  const [chargement,  setChargement]  = useState(true)
  const [erreur,      setErreur]      = useState(null)

  // ─────────────────────────────────────────────────────────
  // useEffect avec [] = s'exécute UNE SEULE FOIS au montage.
  // C'est l'équivalent de "componentDidMount" en React classique.
  // ─────────────────────────────────────────────────────────
  useEffect(() => {

    // On ne peut pas mettre async directement sur la callback de
    // useEffect. On déclare donc une fonction async à l'intérieur
    // et on l'appelle immédiatement.
    const charger = async () => {
      try {
        // fetch() envoie une requête HTTP.
        // await attend la réponse avant de continuer.
        // Sans await, le code passerait à la ligne suivante
        // avant d'avoir reçu les données → bug.
        const reponse = await fetch('/api/ateliers')

        // reponse.json() lit le corps de la réponse et le convertit
        // en objet JavaScript (le JSON texte → objet JS)
        const donnees = await reponse.json()

        setAteliers(donnees)    // on stocke les données dans le state
        setChargement(false)   // on arrête le spinner de chargement

      } catch (e) {
        // Si fetch échoue (réseau coupé, serveur erreur...),
        // on attrape l'erreur et on l'affiche à l'utilisateur.
        setErreur(e.message)
        setChargement(false)
      }
    }

    charger() // on appelle la fonction async définie juste au-dessus

  }, []) // ← tableau vide : ne se déclenche qu'une fois au montage


  // ─────────────────────────────────────────────────────────
  // On gère les 3 états AVANT le return principal.
  // Ces "return anticipés" évitent d'imbriquer des conditions.
  // ─────────────────────────────────────────────────────────
  if (chargement) return <p>Chargement en cours...</p>
  if (erreur)     return <p style={{ color: 'red' }}>Erreur : { erreur }</p>

  return (
    <ul>
      {ateliers.map(atelier => (
        <li key={atelier.id}>{atelier.titre}</li>
      ))}
    </ul>
  )
}
Minuteur.jsx — cleanup (nettoyage) indispensable 
import { useState, useEffect } from 'react'

function Minuteur() {
  const [secondes, setSecondes] = useState(0)

  useEffect(() => {
    // setInterval exécute une fonction toutes les X millisecondes.
    // 1000ms = 1 seconde.
    // On stocke l'identifiant du timer dans "idTimer"
    // pour pouvoir l'arrêter plus tard.
    const idTimer = setInterval(() => {
      // Forme fonctionnelle du setter : quand la nouvelle valeur
      // dépend de l'ancienne, on passe une fonction plutôt que
      // la valeur directement. Ici "s" est la valeur actuelle.
      // Ça évite les problèmes de closures stales.
      setSecondes(s => s + 1)
    }, 1000)

    // ─────────────────────────────────────────────────────
    // FONCTION DE CLEANUP (nettoyage) :
    // useEffect peut retourner une fonction.
    // Cette fonction est appelée automatiquement quand :
    //   • Le composant disparaît de l'écran (démontage)
    //   • Juste AVANT que l'effet se ré-exécute
    //
    // Sans ce cleanup, le timer continuerait à tourner en
    // arrière-plan même si le composant n'est plus affiché.
    // C'est une fuite mémoire → bug difficile à détecter.
    // ─────────────────────────────────────────────────────
    return () => {
      clearInterval(idTimer) // on arrête le timer proprement
    }

  }, []) // [] = démarre le timer une fois au montage, cleanup au démontage

  return <p>Temps écoulé : { secondes } secondes</p>
}
05

Gestion des événements

Les événements sont les interactions de l'utilisateur avec l'interface : il clique, tape au clavier, survole, soumet un formulaire... React les gère via des props spéciales en camelCase (onClick, onChange, onSubmit...) auxquelles on passe une fonction.

  HTML CLASSIQUE vs REACT :

  HTML :    <button onclick="maFonction()">
  React :   <button onClick={maFonction}>
                        ↑         ↑
                    camelCase  référence (pas d'appel)

  RÉFÉRENCE vs APPEL — différence critique :

  onClick={maFonction}    ← CORRECT : on passe la référence
                            React l'appellera au moment du clic

  onClick={maFonction()}  ← INCORRECT : on appelle la fonction
                            immédiatement lors du rendu !
                            (résultat : maFonction s'exécute
                            à chaque affichage, pas au clic)

  TABLEAU DES ÉVÉNEMENTS COURANTS :
  ┌─────────────────┬────────────────────────────────────────┐
  │ onClick         │ Clic sur un élément                    │
  │ onChange        │ Changement dans un input/select        │
  │ onSubmit        │ Soumission d'un <form>                 │
  │ onKeyDown       │ Touche enfoncée                        │
  │ onKeyUp         │ Touche relâchée                        │
  │ onFocus         │ Élément gagne le focus                 │
  │ onBlur          │ Élément perd le focus                  │
  │ onMouseEnter    │ Souris entre dans l'élément            │
  │ onMouseLeave    │ Souris sort de l'élément               │
  │ onScroll        │ Scroll dans un élément                 │
  └─────────────────┴────────────────────────────────────────┘
GestionEvenements.jsx — tous les cas expliqués 
import { useState } from 'react'

function GestionEvenements() {
  const [texte, setTexte] = useState('')

  // ─── GESTIONNAIRE DÉFINI EN DEHORS DU JSX ────────────────
  // Bonne pratique : nommer les fonctions et les sortir du JSX
  // pour garder le JSX lisible. Nom en français avec "gerer" au début.
  // ────────────────────────────────────────────────────────
  const gererClic = () => {
    alert('Bouton cliqué !')
  }

  // ─── ÉVÉNEMENT AVEC OBJET EVENT ──────────────────────────
  // "e" (ou "event") est passé automatiquement par React.
  // Il contient des infos sur ce qui s'est passé :
  //   e.target        → l'élément HTML déclencheur
  //   e.target.value  → valeur actuelle de l'input
  //   e.target.name   → attribut name de l'input
  //   e.key           → touche appuyée (pour onKeyDown)
  // ────────────────────────────────────────────────────────
  const gererChangement = (e) => {
    setTexte(e.target.value)
  }

  // ─── SOUMISSION DE FORMULAIRE ──────────────────────────
  // TRÈS IMPORTANT : e.preventDefault() empêche le comportement
  // par défaut du navigateur. Sans ça, la soumission d'un form
  // RECHARGE la page entière → on perd tout le state React !
  // ────────────────────────────────────────────────────────
  const gererSoumission = (e) => {
    e.preventDefault() // OBLIGATOIRE pour ne pas recharger la page
    console.log('Formulaire soumis avec :', texte)
  }

  // ─── PASSER DES ARGUMENTS SUPPLÉMENTAIRES ────────────────
  // Si tu as besoin de passer des données en plus de l'événement,
  // tu enveloppes dans une fonction fléchée.
  // ────────────────────────────────────────────────────────
  const gererClicAvecId = (id, e) => {
    e.stopPropagation() // empêche la propagation vers les parents
    console.log('Atelier cliqué, id :', id)
  }

  return (
    <div>

      {/* Gestionnaire externe (propre) */}
      <button onClick={gererClic}>
        Cliquer
      </button>

      {/* Inline (acceptable pour du simple) */}
      <button onClick={() => console.log('ok')}>
        Inline
      </button>

      {/* Avec argument supplémentaire : envelopper dans une flèche */}
      <button onClick={(e) => gererClicAvecId('workshop-1', e)}>
        Atelier 1
      </button>

      {/* onChange sur un input → e.target.value donne la valeur tapée */}
      <input
        type="text"
        value={texte}
        onChange={gererChangement}
        placeholder="Taper quelque chose"
      />

      {/* onSubmit sur le form → pas sur le bouton */}
      <form onSubmit={gererSoumission}>
        <button type="submit">Envoyer</button>
      </form>

    </div>
  )
}

Rendu conditionnel & Listes

Rendu conditionnel : afficher un élément seulement si une condition est vraie. Ex : afficher un message d'erreur seulement s'il y a une erreur.

Rendu de liste : transformer un tableau de données en liste d'éléments JSX. La méthode JavaScript .map() est la pierre angulaire de ce mécanisme.

  LES 3 SYNTAXES CONDITIONNELLES :

  1. if/else — AVANT le return (pour les gros blocs)
  ──────────────────────────────────────────────────
  if (chargement) return <Spinner />
  if (!utilisateur) return <PageConnexion />
  return <TableauDeBord />

  2. Ternaire — DANS le JSX (pour alterner entre 2 éléments)
  ──────────────────────────────────────────────────────────
  { estConnecte
    ? <p>Bienvenue !</p>
    : <p>Veuillez vous connecter</p>
  }

  3. && (ET logique) — DANS le JSX (pour afficher OU rien)
  ────────────────────────────────────────────────────────
  { erreur && <MessageErreur texte={erreur} /> }
  ↑ Si erreur est truthy → affiche MessageErreur
  ↑ Si erreur est null/undefined → n'affiche rien

  ⚠️ PIÈGE avec && : utiliser Boolean() ou !! pour les nombres :
  { nbItems && <Liste /> }  → si nbItems = 0, affiche "0" dans le DOM !
  { nbItems > 0 && <Liste /> }  → correct


  .map() — TRANSFORMER un tableau en JSX :

  const fruits = ['pomme', 'poire', 'cerise']
                       │
                       ▼ .map() parcourt chaque élément
  fruits.map(fruit => <li key={fruit}>{fruit}</li>)
                    ↑
                    Une fonction qui reçoit l'élément actuel
                    et retourne du JSX
                       │
                       ▼ résultat : tableau de JSX
  [ <li key="pomme">pomme</li>,
    <li key="poire">poire</li>,
    <li key="cerise">cerise</li> ]

  React affiche ce tableau directement dans le DOM.
PageAteliers.jsx — conditionnel et liste complètes 
function PageAteliers({ chargement, erreur, ateliers }) {

  // ─── CAS 1 : SORTIE ANTICIPÉE ────────────────────────────
  // Ces if/return avant le JSX principal évitent d'imbriquer
  // des conditions. Le code reste plat et lisible.
  // Dès qu'une condition est vraie, on retourne immédiatement.
  // ────────────────────────────────────────────────────────
  if (chargement) {
    return <p>Chargement...</p>
  }

  if (erreur) {
    return <p className="erreur">Erreur : { erreur }</p>
  }

  // Si on arrive ici, chargement=false et erreur=null

  return (
    <div>

      {/* ─── CAS 2 : TERNAIRE DANS LE JSX ─────────────────────
           Condition ? si_vrai : si_faux
           Utile pour afficher l'un ou l'autre selon l'état */}
      { ateliers.length > 0
        ? <p>{ ateliers.length } atelier(s) disponible(s)</p>
        : <p>Aucun atelier pour le moment.</p>
      }

      {/* ─── CAS 3 : && POUR AFFICHER OU RIEN ─────────────────
           Si ateliers.length > 0 est VRAI → affiche le <ul>
           Si c'est FAUX → n'affiche rien (pas même un nœud vide) */}
      { ateliers.length > 0 &&
        <ul className="liste-ateliers">

          {/* ─── .map() POUR RENDRE UNE LISTE ─────────────────────
               ateliers est un tableau d'objets.
               .map() retourne un nouveau tableau de JSX.
               React affiche ce tableau comme une liste d'éléments.

               La prop "key" EST OBLIGATOIRE dans un .map().
               Elle doit être :
               • unique parmi les éléments du même niveau
               • stable (ne doit pas changer entre les rendus)
               Utilise toujours l'id de ta donnée. JAMAIS l'index ! */}
          { ateliers.map((atelier) => (
            <li key={atelier.id}>

              {/* On accède aux propriétés de chaque objet atelier */}
              <strong>{ atelier.titre }</strong>
              <span> — { atelier.niveau }</span>

              {/* Condition dans la liste : badge seulement si complet */}
              { atelier.currentParticipants === atelier.maxParticipants
                && <span className="badge-complet">Complet</span>
              }

            </li>
          )) }

        </ul>
      }

    </div>
  )
}

React Router v6 — Routes dans leur propre fichier

Pourquoi un router ?
Une application React est une SPA (Single Page Application) : il y a un seul fichier HTML, mais plusieurs "vues" (pages). React Router intercepte les changements d'URL et affiche le bon composant sans recharger la page.

Pourquoi un fichier séparé pour les routes ?
Si tu mets les routes dans App.jsx, ce fichier grossit rapidement et mélange deux responsabilités (layout + navigation). Le fichier router.jsx a une seule responsabilité : définir les routes. C'est une meilleure architecture.

  STRUCTURE AVEC ROUTER SÉPARÉ :

  src/
  ├── main.jsx          ← point d'entrée, importe AppRouter
  ├── router.jsx        ← TOUTES les routes sont ici
  ├── AppLayout.jsx     ← layout partagé (navbar + <Outlet />)
  │
  ├── pages/
  │   ├── Accueil.jsx
  │   ├── Ateliers.jsx
  │   └── DetailAtelier.jsx
  │
  └── composants/
      └── Navigation.jsx


  COMMENT createBrowserRouter FONCTIONNE :

  On lui passe un tableau d'objets, chaque objet est une route.
  La route "/" a des "children" : ce sont les sous-routes.
  Le composant AppLayout s'affiche toujours.
  L'enfant actif s'affiche à la place de <Outlet />.

  URL : /ateliers
        │
        ▼
  BrowserRouter intercepte le changement d'URL
        │
        ▼
  Cherche dans les routes : path="ateliers" → trouvé !
        │
        ▼
  Affiche :  <AppLayout>
               <Navigation />        ← toujours visible
               <Outlet />            ← remplacé par <Ateliers />
             </AppLayout>


  HOOKS DU ROUTER :

  useNavigate()  → fonction pour changer de page par code
  useParams()    → lire les :paramètres de l'URL
  useLocation()  → lire l'URL actuelle complète
  NavLink        → lien avec classe "active" automatique
  Outlet         → emplacement de la page enfant courante
router.jsx — fichier dédié, toutes les routes 
// createBrowserRouter : crée le routeur avec une configuration d'objet.
// C'est la méthode moderne (React Router v6.4+).
// Elle permet aussi de faire du chargement de données (loaders),
// mais on ne l'utilise pas ici pour rester simple.
//
// RouterProvider : composant React qui "fournit" le routeur
// à toute l'application. On l'utilise dans main.jsx.
import { createBrowserRouter, RouterProvider } from 'react-router-dom'

// On importe le layout partagé et toutes les pages
import AppLayout       from './AppLayout'
import TableauDeBord   from './pages/TableauDeBord'
import TousLesAteliers from './pages/TousLesAteliers'
import DetailAtelier   from './pages/DetailAtelier'
import MesAteliers     from './pages/MesAteliers'
import EspaceFormateur from './pages/EspaceFormateur'

// ────────────────────────────────────────────────────────────────
// createBrowserRouter reçoit un tableau de configurations de routes.
// Chaque objet représente une route.
// ────────────────────────────────────────────────────────────────
const routeur = createBrowserRouter([
  {
    // path: '/' → correspond à l'URL de base (ex: http://localhost:5173/)
    // Ce niveau est le layout partagé. Il s'affiche toujours,
    // quelle que soit la sous-page visitée.
    path: '/',
    element: <AppLayout />,

    // children : les routes enfants qui s'affichent à l'intérieur
    // du <Outlet /> de AppLayout.
    children: [
      {
        // index: true → cette route correspond exactement à "/"
        // sans segment supplémentaire. C'est la page d'accueil.
        index: true,
        element: <TableauDeBord />
      },
      {
        // path: 'ateliers' → correspond à /ateliers
        // Pas besoin du "/" au début pour les enfants.
        path: 'ateliers',
        element: <TousLesAteliers />
      },
      {
        // ':id' est un paramètre dynamique.
        // Il correspond à n'importe quelle valeur après /ateliers/
        // Ex: /ateliers/workshop-1  →  id = "workshop-1"
        //     /ateliers/abc123      →  id = "abc123"
        // On récupère sa valeur avec useParams() dans le composant.
        path: 'ateliers/:id',
        element: <DetailAtelier />
      },
      {
        path: 'mes-ateliers',
        element: <MesAteliers />
      },
      {
        path: 'espace-formateur',
        element: <EspaceFormateur />
      },
      {
        // '*' correspond à TOUT ce qui ne matche aucune route ci-dessus.
        // On redirige vers "/" pour éviter une page blanche.
        path: '*',
        element: <TableauDeBord />
      }
    ]
  }
])

// Ce composant est ce qu'on appelle dans main.jsx.
// RouterProvider reçoit le routeur configuré et injecte
// le contexte du router dans toute l'application.
function AppRouter() {
  return <RouterProvider router={routeur} />
}

export default AppRouter

// useParams : hook pour lire les paramètres dynamiques de l'URL.
//             Si la route est "ateliers/:id" et l'URL est "/ateliers/workshop-3",
//             useParams() retourne l'objet { id: "workshop-3" }.
//
// useNavigate : hook qui retourne une fonction pour naviguer par code.
//              Utile pour rediriger après une action (ex: soumission de form).
import { useParams, useNavigate } from 'react-router-dom'
import { utiliserSkillHub } from '../contextes/ContexteSkillHub'

function DetailAtelier() {

  // On destructure { id } depuis useParams().
  // "id" correspond au ":id" défini dans la route.
  // Si la route était "/:categorie/:id", on aurait { categorie, id }.
  const { id } = useParams()

  // useNavigate retourne une fonction "naviguer".
  // On peut lui passer :
  //   • un chemin string : naviguer('/ateliers')
  //   • un nombre :        naviguer(-1) = page précédente
  //                        naviguer(1)  = page suivante (si disponible)
  const naviguer = useNavigate()

  // On cherche l'atelier correspondant à l'id dans notre contexte
  const { ateliers, sInscrire, seDesinscrire } = utiliserSkillHub()
  const atelier = ateliers.find(a => a.id === id)

  // Sortie anticipée si l'atelier n'existe pas
  if (!atelier) {
    return (
      <div>
        <p>Atelier introuvable.</p>
        <button onClick={() => naviguer('/ateliers')}>
          Retour aux ateliers
        </button>
      </div>
    )
  }

  return (
    <div>
      {/* naviguer(-1) = équivalent du bouton "retour" du navigateur */}
      <button onClick={() => naviguer(-1)}>← Retour</button>

      <h1>{ atelier.titre }</h1>
      <p>{ atelier.description }</p>

      <button
        onClick={() => atelier.estInscrit
          ? seDesinscrire(atelier.id)
          : sInscrire(atelier.id)
        }
      >
        { atelier.estInscrit ? 'Se désinscrire' : "S'inscrire" }
      </button>
    </div>
  )
}

export default DetailAtelier

Context API

Le problème du prop drilling :
Tu as l'utilisateur connecté dans ton composant racine. Un composant Avatar au fond de l'arbre en a besoin. Sans Context, tu dois passer l'utilisateur en prop à chaque niveau intermédiaire, même si ces niveaux n'en ont pas besoin. C'est pénible et fragile.

La solution Context :
Le contexte crée une "télécabine" qui transporte les données directement du fournisseur (Provider) jusqu'aux consommateurs (useContext), sans passer par les intermédiaires.

  SANS CONTEXT (prop drilling) :         AVEC CONTEXT :

  App (a utilisateur)                    App
   └─ Page (reçoit utilisateur)    ┌──── Provider (fournit utilisateur)
       └─ Section (reçoit utilis.) │      │
           └─ Carte (reçoit utilis)│    Page  Section  Carte  Avatar
               └─ Avatar           │                           │
                  UTILISE enfin !  │                    useContext()
                                   │                    ACCÈS DIRECT ↑
                                   └─ la "télécabine"


  LES 3 ÉTAPES :

  1. createContext()   → crée la "boîte" de contexte
  2. Provider          → "remplit" la boîte et l'injecte dans l'arbre
  3. useContext()       → n'importe quel enfant lit la boîte
contextes/ContexteSkillHub.jsx — créer le contexte 
import { createContext, useContext, useState } from 'react'
import { mockAteliers, mockUtilisateur }        from '../données/mockData'

// ─── ÉTAPE 1 : createContext ──────────────────────────────────
// createContext() crée un objet contexte.
// On lui passe null comme valeur par défaut (quand on est
// en dehors du Provider, ce qui ne devrait pas arriver).
// On stocke ce contexte dans une variable avec une majuscule
// car c'est une "classe" de contexte React. ─────────────────────
const ContexteSkillHub = createContext(null)

// ─── ÉTAPE 2 : FournisseurSkillHub (le Provider) ─────────────
// Ce composant enveloppe toute l'app (dans main.jsx).
// Il contient les états globaux et les fonctions pour les modifier.
// Tout ce qu'on met dans "value" est accessible partout dans l'app.
// ────────────────────────────────────────────────────────────────
export function FournisseurSkillHub({ children }) {

  // Le state est ici, dans le Provider.
  // Quand il change, tous les composants consommateurs re-rendent.
  const [ateliers,    setAteliers]    = useState(mockAteliers)
  const [utilisateur, setUtilisateur] = useState(mockUtilisateur)

  // Fonction pour s'inscrire à un atelier.
  // Elle met à jour DEUX states en même temps :
  //   1. L'atelier : on incrémente currentParticipants et on marque estInscrit
  //   2. L'utilisateur : on ajoute l'id de l'atelier à sa liste
  const sInscrire = (id) => {
    // .map() pour mettre à jour un élément dans le tableau
    // sans modifier le tableau original (immutabilité !).
    // Pour chaque atelier, si son id correspond → on le modifie.
    // Sinon → on le retourne tel quel.
    setAteliers(precedents =>
      precedents.map(a =>
        a.id === id
          ? { ...a, estInscrit: true, currentParticipants: a.currentParticipants + 1 }
          : a
      )
    )
    setUtilisateur(precedent => ({
      ...precedent,
      ateliersInscrits: [...precedent.ateliersInscrits, id]
    }))
  }

  const seDesinscrire = (id) => {
    setAteliers(precedents =>
      precedents.map(a =>
        a.id === id
          ? { ...a, estInscrit: false, currentParticipants: a.currentParticipants - 1 }
          : a
      )
    )
    setUtilisateur(precedent => ({
      ...precedent,
      // .filter() retourne un nouveau tableau sans l'élément avec cet id
      ateliersInscrits: precedent.ateliersInscrits.filter(aid => aid !== id)
    }))
  }

  return (
    // Provider.value = les données et fonctions qu'on expose.
    // Tous les composants enfants pourront y accéder.
    // L'objet dans value doit être stable pour éviter des
    // re-rendus inutiles (sujet avancé, pas critique ici).
    <ContexteSkillHub.Provider
      value={{ ateliers, utilisateur, sInscrire, seDesinscrire }}
    >
      { children }
    </ContexteSkillHub.Provider>
  )
}

// ─── ÉTAPE 3 : hook personnalisé pour consommer le contexte ──
// On crée une fonction utiliserSkillHub() qui encapsule useContext.
// Avantages :
//   • Syntaxe plus courte dans les composants
//   • Vérification si on est bien dans le Provider (throw utile)
//   • Un seul import à faire dans chaque composant consommateur
// ────────────────────────────────────────────────────────────────
export function utiliserSkillHub() {
  const contexte = useContext(ContexteSkillHub)

  // Sécurité : si utiliserSkillHub est appelé hors du Provider,
  // on lève une erreur claire plutôt qu'un bug mystérieux.
  if (!contexte) {
    throw new Error('utiliserSkillHub doit être appelé dans FournisseurSkillHub')
  }

  return contexte
}
